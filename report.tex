\begin{center}
{\Large Исследование строковых сортировок}
\end{center}

\bigskip

В этой работе я сравнивал обычные алгоритмы сортировки строк и специальные алгоритмы, которые лучше подходят именно для строк.

Обычные сортировки хорошо работают на числах и объектах, которые можно сравнивать за постоянное время.  
Но для строк ситуация другая: сравнение строк обычно идёт посимвольно. Если строки имеют длинные общие префиксы, то даже одно сравнение может стоить довольно дорого. Поэтому хотелось посмотреть, как на практике ведут себя обычные сортировки и специальные строковые алгоритмы.

\bigskip

\textbf{Что сравнивалось}

\begin{itemize}
    \item обычный QuickSort;
    \item обычный MergeSort;
    \item ternary String QuickSort;
    \item String MergeSort;
    \item MSD Radix Sort;
    \item MSD Radix Sort с переключением на String QuickSort для маленьких подмассивов.
\end{itemize}

Для каждого алгоритма измерялись:
\begin{itemize}
    \item время работы;
    \item количество посимвольных сравнений.
\end{itemize}

\bigskip

\textbf{Генерация данных}

Для подготовки тестов был сделан класс \texttt{StringGenerator}.  
Он создаёт массивы строк из символов следующего алфавита:
\begin{itemize}
    \item заглавные латинские буквы;
    \item строчные латинские буквы;
    \item цифры;
    \item специальные символы.
\end{itemize}

Длина строк выбиралась от 10 до 200 символов.  
Размеры массивов менялись от 100 до 3000 с шагом 100.

Использовались такие типы массивов:
\begin{itemize}
    \item случайные;
    \item обратно отсортированные;
    \item почти отсортированные;
    \item массивы с длинными общими префиксами.
\end{itemize}

Почти отсортированные массивы получались из отсортированного массива с помощью небольшого числа случайных перестановок.  
Для массивов с общими префиксами специально генерировались строки, у которых начало совпадает, а дальше уже идут случайные символы.

\bigskip

\textbf{Как проводились измерения}

Для измерений был сделан класс \texttt{StringSortTester}.  
Он запускал сортировки на одинаковых входных данных и сохранял результаты в CSV-файл.

Времена замерялись в миллисекундах.  
Также отдельно считалось число посимвольных сравнений, потому что именно оно хорошо показывает, насколько алгоритм эффективно работает именно на строках.

Результаты сохранялись в файл \texttt{data/results.csv}, а потом по ним строились графики.

\bigskip

\textbf{Ожидания перед экспериментом}

От обычных QuickSort и MergeSort можно ожидать, что они будут вести себя более-менее нормально на случайных строках, но на данных с длинными общими префиксами начнут проигрывать. Причина в том, что такие алгоритмы не используют уже найденную информацию о совпадающих началах строк и могут много раз заново сравнивать одни и те же символы.

От String QuickSort ожидалось хорошее поведение именно на строках с общими префиксами, так как он работает посимвольно и делит строки по текущему символу.

От String MergeSort ожидалось улучшение по числу посимвольных сравнений по сравнению с обычным MergeSort.

От MSD Radix Sort ожидалось хорошее время на больших массивах строк, особенно когда строки достаточно длинные.

От версии MSD Radix Sort с переключением на String QuickSort ожидалось, что она будет работать стабильнее, чем чистый MSD Radix Sort, так как на маленьких подмассивах служебные расходы radix-sort уже начинают мешать.

\bigskip

\textbf{Что видно по результатам}

По случайным данным обычные алгоритмы ведут себя вполне ожидаемо.  
Разница между стандартными и адаптированными сортировками есть, но она не всегда очень большая.

На обратно отсортированных и почти отсортированных массивах уже лучше видно разницу между подходами.  
Особенно интересно было смотреть на число посимвольных сравнений: у строковых алгоритмов оно обычно меньше, чем у обычных QuickSort и MergeSort.

Самая заметная разница проявляется на массивах с длинными общими префиксами.  
Именно там обычные сортировки начинают делать много лишней работы, потому что одинаковые начала строк заново сравниваются очень много раз.  
Специальные строковые алгоритмы работают на таких данных лучше.

По результатам видно, что адаптированные алгоритмы действительно оправдывают ожидания.  
Особенно хорошо это заметно у String QuickSort, String MergeSort и MSD Radix Sort с переключением.

\bigskip

\textbf{Сравнение с теорией}

Полученные результаты в целом соответствуют тому, что ожидалось из теории.

Обычные сортировки на сравнениях имеют хорошую асимптотику по числу сравнений, но для строк этого уже недостаточно, потому что каждое сравнение может быть дорогим.

String MergeSort и String QuickSort лучше используют информацию о префиксах, поэтому уменьшают количество лишних посимвольных сравнений.

MSD Radix Sort вообще работает по символам и не строится на обычных строковых сравнениях, поэтому на длинных строках и больших массивах показывает хорошие результаты.

Версия MSD Radix Sort с переключением на String QuickSort выглядит наиболее практичной, потому что убирает лишние расходы на маленьких подмассивах.

\bigskip

\textbf{Вывод}

По итогам эксперимента можно сделать такой вывод:  
если сортируются строки, особенно похожие между собой, то обычные QuickSort и MergeSort перестают быть лучшим выбором. На таких данных специальные строковые алгоритмы работают заметно лучше, особенно по числу посимвольных сравнений.

Наиболее интересными по результатам оказались:
\begin{itemize}
    \item String QuickSort;
    \item String MergeSort;
    \item MSD Radix Sort с переключением на String QuickSort.
\end{itemize}

То есть общий вывод довольно простой: для строк действительно стоит использовать специальные алгоритмы, а не только обычные сортировки сравнениями.
